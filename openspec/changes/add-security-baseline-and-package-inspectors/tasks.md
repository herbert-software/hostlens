## 2. 包管理域（builtin/pkg/）

- [x] 2.1 `builtin/pkg/pending_updates.yaml`（`pkg.pending_updates`）：collector 内 `command -v apt-get`/`dnf` 自适应分支（D-1，**else 分支 `echo>&2; exit 1` fail-loud**）→ 可升级包计数（apt 精确 `grep -c '^Inst'` 非 wc -l，Failure Mode 4）；**pipe-safe（D-6.1：先 `raw=$(apt-get -s upgrade) || exit 1` 再 grep，禁裸管道）；apt/dnf 分支内主命令失败也 fail-loud（D-6.4）；dnf 分支用 `dnf check-update` 故豁免 exit 100（D-6.3）**；`requires_binaries` 只放 `[awk]`/`[grep]`，**不放 apt-get/dnf、不放 sh**（D-1）。验收：snapshot 含 apt 多包 + dnf 多包（双分支）+ 无包管理器（exception）+ **「包管理器在但主命令失败（mock apt-get 返非零）→ exception」fixture（D-6.4，非假 0）**
- [x] 2.2 `builtin/pkg/security_patches.yaml`（`pkg.security_patches`）：apt（security 源过滤）/`dnf updateinfo -q list security` → 待打安全补丁计数；apt/dnf 分支 + pipe-safe + fail-loud（D-6.1/6.4；**`dnf updateinfo` 不返 exit 100，走标准 `|| exit 1`、不套 D-6.3 的 100-豁免**）。**过滤器错配防护**：apt 的 security 源过滤正则错写会令「有补丁」假 0（与 D-6.5 unit 名假阴同构）——验收含「**post-filter 计数非 0 → 必须检出 finding**」finding-trigger fixture，锁住**下游计数 + finding 触发链**；**过滤器 regex 本身在 D-7 offline 不执行（fixture 录 collector 最终 JSON），不被 offline fixture 锁——regex 命令串级锁 + 真机 Demo Path 验证计数边界**（见偏离登记）。验收：snapshot 含有安全补丁待打（finding-trigger，锁下游计数+finding）+ 无补丁（ok）+ 无包管理器（exception）+ 主命令失败（exception）
- [x] 2.3 `builtin/pkg/held_back.yaml`（`pkg.held_back`）：`apt-mark showhold`/dnf versionlock → 被 hold/pin 挡住升级的包列表。**`apt-mark` 空输出有歧义（无 held vs 命令失败）——必须捕获退出码判定（D-6.4），不靠空输出推断成功**。**归档前锁定单一名**（held_back，不留 `last_update_age` 备选），fixture/snapshot 按锁定名命名。验收：snapshot 含有 held 包（finding）+ 无 held（ok）+ 无包管理器（exception）+ 命令失败（exception）

## 3. Fixture 录制与 snapshot 测试

- [x] 3.1 用**既有 os-shell 录制约定（design D-7 / `_CaptureTarget` 模式，见 `tests/inspectors/_record_os_disk_fs.py`/`_record_os_compute_memory.py`）**为 6 个 inspector 录 ReplayTarget fixture，落 `tests/inspectors/fixtures/security/`、`fixtures/pkg/`。约定：写 `tests/inspectors/_record_security_pkg.py` 驱动**真** `InspectorRunner`，**捕获真实渲染的命令串**（字节级匹配防 fixture-runner 漂移，Failure Mode 5），**每场景的 stdout/exit_code 由作者编（代表该状态下 collector 会吐的输出）**——这是项目对 Linux-only os-shell inspector 的既有 SOT 约定，与 wave-1 同类一致，**不用 docker/真 Linux**。`_CaptureTarget` 需扩展支持 **exit_code≠0 + 空 stdout** 场景（exception：collector fail-loud 后 runner 见非零退出 + 空 stdout → JSONDecodeError → status=exception）。
  > **review 偏离登记（D-7 架构下「shell 逻辑只命令串级锁、不行为级执行」的统一清单）**：review-loop 曾把本任务硬化为「真机录制、RHEL 机录 unit-OR」。实现期发现项目已有 D-7 约定（命令串真捕获防漂移 + 场景 stdout 作者编），与 wave-1 同类一致，且 collector 内的 shell 逻辑在 ReplayTarget 架构下本不可执行（_CaptureTarget 对整条 collect.command 返回作者编的 stdout，不跑真 shell）。**以下 collector-shell 正确性点因此只由命令串捕获 + 真机 Demo Path 锁定、不由 offline snapshot 行为级验证**，已与用户确认改回 D-7：
  > - **D-6.5 unit-OR**（failed_logins 双 `_SYSTEMD_UNIT`）：journalctl OR 语义不跑；命令串确含两 unit（有测试断言），下游检出由 fixture 证。
  > - **D-6.2 find-rc 第三态**（world_writable「部分子树拒绝 rc≠0 但有数据→ok」）：在 replay 边界与「正常有可写目录」**字节相同**（都 exit0+JSON，D-6.2 判据在 collect.command 内已解析），**架构上无法造出 distinct fixture**；find-rc 判据命令串级锁。
  > - **过滤器正确性**（security_patches `-security` 过滤、pending_updates `^Inst` 计数）：fixture 录的是 collector **最终 JSON**（post-filter count），过滤器 regex 不跑——broken filter 不被 offline fixture 抓住；regex 命令串级锁 + 真机 Demo Path 验证计数边界。
  > - 通用：D-6.1 pipe-safe / D-6.3 dnf-100 / D-6.4 分支内 fail-loud 同属命令串级锁（shell 逻辑经 review 的 live shell 实测确认正确，但不在 offline CI 执行）。
  > 行为级真验证留**真机 Demo Path**（带 journald/apt/dnf 的 host 跑一次确认计数边界与过滤器）。manifest/test 注释已下调措辞、不夸大 offline fixture 的锁定范围。
