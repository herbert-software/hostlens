## 为什么

2026-06-24 08:00 的 `daily-health-fleet` 巡检产出 `partial`：6 台机里 cloudcone 的前 5 个 inspector 全 `target_unreachable`（`ssh_connect_timeout`），但**同一次 run 里该机后 3 个 inspector 又成功了**（连接在 ~70s 时建好、被缓存复用）。即 cloudcone 没宕，只是 Tailscale 冷路径首次建连耗时 >10s；先跑的 inspector 撞上冷连接被记成失败，后跑的蹭到已建立的连接就过了。推送只显示「5 项失败」的数字、不说哪台/为什么，运维无从判断这是真故障还是抖动。

根因在 `ssh-execution-target` 契约：现有规范**显式禁止首次 connect 重试**（「重连流程仅适用于已建立连接断开场景、禁止用于首次 connect」）。该契约当初假设「首连要么 ~10s 内成功、要么 host 宕」，只有两态。真实 Tailscale fleet 暴露了第三态：**可达但冷路径需 ~70s 建连**（节点空闲后需重新 DERP→direct / wireguard 握手）。`connect_timeout` 只是单次尝试的上限，对「会逐步建立」的路径无论设多大都是赌博；正确解是**在一个总预算内反复尝试**。

## 变更内容

- **`SSHTarget` 首次 connect 在预算内重试（构造参数，按调用路径 opt-in）**：`SSHTarget` 加构造参数 `cold_connect_retry_budget_seconds: float | None`（**默认 `None` = 单次尝试 = 完全保持现状**）。非 None 时，首次建连在该预算内反复 `asyncssh.connect`，直到成功或预算耗尽。**预算是构造参数、不是 per-target entry 字段**——只有两条路径在构造 `SSHTarget` 时传非 None 预算（硬默认 `_DEFAULT_COLD_CONNECT_RETRY_BUDGET_SECONDS = 90.0`）：① **定时 fleet 巡检**（`cli/schedule.py` 的调度 registry 构造）；② **纳管探活** `target import` / `propose_target_import`（经 `registry.build_one_target`——该 helper 在 `targets/registry.py`、非 `probe.py`，`probe.py` 只是调用点；helper 自身加参透传其内部 `build_registry_from_config`，纳管新机本就可能冷）。**doctor（`cli/doctor.py`）/ `target test`·`list`（`cli/target.py`）/ 临时 `inspect`（`cli/inspect.py`）/ `mcp serve`（`cli/mcp.py`）/ `fix`（`cli/fix.py`）/ demo（`demo/assembly.py`，replay-only 永不 dial）** 的 registry 构造一律**不**传预算（默认 `None` → 保持各自 5–12s 快速失败的响应契约，零行为变更）。
- **只重试网络瞬时类、且每次尝试 cap 到剩余预算**：重试**仅**捕获 `ssh_connect_timeout`（`TimeoutError`/`OSError`/`socket.gaierror`/`ConnectionRefused` 归类，真正的网络瞬时态）→ 重算剩余预算，**`<=0` 不再退避直接 stamp+raise，否则 `sleep(min(1.0, 剩余预算))`** 后重试。**`ssh_auth_failed`（认证/host-key/KEX）与 `ssh_connect_failed`（asyncssh 未分类兜底）一律不重试、立即上抛**——后者兜底里可能含 host-key 不匹配等永久错误，盲目重试 = 把改过的 host key 当瞬时态反复探测（安全隐患）。每次尝试的 `connect_timeout` **cap 到 `min(connect_timeout, 剩余预算)`**（需给 `_open_connection`/`_connect_kwargs` 加 timeout override 参，否则 cap 静默失效），否则贴着截止线开始的最后一次尝试会超出预算整整一个 `connect_timeout`；退避也界住截止线，故总耗时 ≤ budget + 末次尝试、不额外加一整个退避。预算耗尽 → 仍 raise `ssh_connect_timeout`（kind 对调用方不变）。
- **冷连接失败的批内负缓存短路（仅对共享的巡检 target）**：定时巡检里一台机的 N 个 inspector **共享同一个长生命周期 `SSHTarget`**（`context_factory` 闭包共享 `TargetRegistry`，跨 run 复用），并发争抢连接锁。若首次冷连接耗尽预算失败，stamp `self._cold_connect_failed_at = time.monotonic()`；锁体内**在 idle sweep 之后、`self._conn is None` 的开连分支之前**检查：stamp 在短 TTL（默认 120s）内 → 立即 fast-fail（不再付预算），防止真宕机器让 N 个 inspector 各付一轮 90s。stamp **只在预算耗尽时写一次**（非每次尝试）、**只在锁入口读**（非循环中途）。**清除点唯一：首次 connect 成功（步④）清 stamp**；`_reconnect` **不**清——stamp-set 态恒 `_conn is None` 而 `_reconnect` 恒要求 `_conn` 曾建立（已过步④清），且负缓存读受 `_conn is None` 门控屏蔽任何残留 stamp，最坏一次 TTL 内假 fast-fail，故不为该实际不可达路径加防御性清除（对齐「不写不可能分支兜底」）。TTL 远短于日巡间隔故不污染明天；连续 trigger（< TTL）间真机恢复有一次假阴性，是有意权衡。纳管探活每台**新建** `SSHTarget`（`build_one_target` 后 `aclose`），stamp 不跨 host，故 probe 只吃预算、不享负缓存（每台一次 exec，无 N× 问题，可接受）。
- **推送列出失败检查的目标与原因（reason 先 key on `status`）**：`failed > 0` 时，Telegram / 飞书 Lark 报告新增「失败检查」段，把 `report.inspector_results` 里 `status ∈ _FAILED_STATUSES`（`{timeout, target_unreachable, exception}`，复用 `_filters` 既有闭集）的项**按 `(target_name, 原因 label)` 分组**，列主机 + 中文原因 + inspector 名单。**原因 label 先按 `status`（5 值闭枚举，恒可映射）取桶级中文**（`timeout`→执行超时 / `exception`→采集异常 / `target_unreachable`→不可达）；**仅 `target_unreachable` 再用 `error`（此 status 下 `error == TargetError.kind`，是枚举串）细化**（`ssh_connect_timeout`→连接超时 / `ssh_auth_failed`→认证失败 / …）；未知 kind **回退到 `target_unreachable` 的桶级中文「不可达」**，**不**漏生英文。**关键**：`error` 对 `timeout`/`exception` 是自由文本（`"collect.command exceeded 30 seconds"` / `"parse_failed: …"`，且模型校验器强制非空），故**不能**以 `error` 为主键——否则渲染生英文、且同主机多个 `exception` 因 message 不同被拆成 N 组（破坏「合一组防刷屏」）。消费**已脱敏**报告（`_redact_inspector_result` 保留 `status`/`target_name`/`error`），零模型字段改动，两通道走同一 `_filters` 过滤器。
- **非目标**：**不**加 per-target `TargetEntry` 预算字段（构造参数即可，省去 `SSHEntry`/`__repr__`/`_entry_to_dict`/Protocol 四处接线 + save 往返）；**不**加 `Settings.ssh` 全局预算字段（硬默认 90s，留 ponytail upgrade path）；**不**改 `connect_timeout` 默认；**不**改重连（已建连断开）算法；**doctor / `target test`·`list` / 临时 `inspect` / `mcp serve` / `fix` / demo 不重试**（保持快速失败）；**不**改 `ssh_connect_timeout` / `target_unreachable` kind 命名；**不**在编排层加 per-target preflight、**不**改 scheduler semaphore（接受多冷目标 `N_cold×90s` 最坏吞吐，见 design 决策4 诚实披露）；**不**修复既存 spec/code 漂移（live spec line 22 写 `BadHostKeyError` 而码用 `HostKeyNotVerifiable`；`_reconnect` 兜底 catch `ssh_connect_failed`）——本提案改 retry-timeout-only 已绕开其安全影响，漂移本身作为独立 follow-up。

## 功能 (Capabilities)

### 修改功能

- `ssh-execution-target`: 首次 connect 从「单次尝试 fast-fail」改为「**可选**预算重试」——新增 `SSHTarget` 构造参数 `cold_connect_retry_budget_seconds`（默认 None=保持现状），非 None 时在预算内**仅重试 `ssh_connect_timeout`**（每次尝试 cap `min(connect_timeout, remaining)`），认证/host-key/未分类兜底立即 raise；预算耗尽后批内负缓存短路（TTL 默认 120s，stamp 一次于耗尽、读一次于锁入口、任一连接成功清除）。重连（已建连断开）算法、idle timeout、keepalive、凭据脱敏不变。
- `notifier-telegram`: 段顺序新增「失败检查」段（仅 `failed > 0` 渲染，覆盖行之后、发现之前）——按 `(target_name, 原因 label)` 分组，原因 **先 key on `status`**、`target_unreachable` 再以 `error`(kind) 细化、未知回退桶级中文。既有抬头/覆盖行/发现优先/去重排序/fleet 主机归因/本地时区不变。
- `notifier-lark`: 飞书卡片同构新增「失败检查」节（仅 `failed > 0`），与 Telegram 同信息结构 + 同一 `_filters` 过滤器；既有节布局/签名/`tojson` 转义不变，条件插入保持卡片 JSON 合法（无悬空逗号）。

## 影响

- **代码**：`src/hostlens/targets/ssh.py`——`__init__` 加 `cold_connect_retry_budget_seconds` 参数 + `self._cold_connect_failed_at`；`_ensure_connection` 锁体加（idle sweep→负缓存 fast-fail→预算重试循环〔仅 `ssh_connect_timeout`、每次 cap、退避界住截止线、耗尽 stamp〕）；`_open_connection`/`_connect_kwargs` 加 `connect_timeout` override 参（cap 生效点）；模块常量 `_DEFAULT_COLD_CONNECT_RETRY_BUDGET_SECONDS=90.0` / `_COLD_CONNECT_RETRY_DELAY=1.0` / `_COLD_CONNECT_NEG_TTL=120.0`；stamp 仅步④（首连成功）清，`_reconnect` **不**触碰（实际不可达 + `_conn is None` 门控屏蔽）。`src/hostlens/targets/registry.py`——`build_registry_from_config` 与 `build_one_target` 均加 `cold_connect_retry_budget_seconds: float | None = None` 参数透传给每个 `SSHTarget`。`src/hostlens/cli/schedule.py`（调度 registry 构造）传 `_DEFAULT_...`；`src/hostlens/targets/probe.py` 调 `registry.build_one_target` 处传 `_DEFAULT_...`。**doctor / `target test`·`list` / `inspect` / `mcp serve` / `fix` / demo 的 registry 构造不传（默认 None，零行为变更）**。`src/hostlens/notifiers/_filters.py`——加 `failed_checks` 过滤器（复用 `_FAILED_STATUSES`，按 `(target, status-or-refined-label)` 分组）+ `_FAIL_STATUS_LABELS`（status→中文）与 `_FAIL_KIND_LABELS`（kind→中文，仅 target_unreachable 细化）；两个模板加「失败检查」段（Lark 节自带 trailing 逗号镜像覆盖行）。**无 `targets/config.py` 改动**（不加 entry 字段）。
- **契约**：`ssh-execution-target` 首次 connect 行为按构造参数变更——传 None 的调用方（doctor/test/list/inspect/mcp/fix/demo/默认）**零行为变更**；传预算的调用方（巡检/import）对可达但慢的 host 改为成功、对真宕 host 多花至多预算时长才判失败。调用方契约稳定（仍是 `TargetError(kind="ssh_connect_timeout")`，只是更晚）。
- **依赖**：无新增依赖。
- **性能 / 吞吐**：单冷目标巡检 ≈ 该目标冷路径时长（它本是 long pole），不额外拖长；**多冷目标在 `Semaphore(8)` 下最坏 `N_cold×90s`**（冷目标各占满槽位串行重试，design 决策4 诚实披露，接受不改编排）。负缓存把单台真宕从 `N_inspectors×90s` 收敛回 ~90s。`target import` 批量纳管 down host 每台至多 90s（用户已选择容忍）。无 LLM 调用。
- **测试**：`ssh.py` 单元测试（**非** integration——integration spec 禁 mock asyncssh）：默认 None 仍单次（回归）、预算内第 K+1 次成功、预算耗尽 raise + stamp、每次尝试 cap 不超预算 + 总耗时 ≤ budget+ε、`ssh_auth_failed` 一次即 raise 不重试、负缓存兄弟 fast-fail（connect 计数不增）、TTL 过期重试。notifier 测试：「失败检查」按 status 桶分组（`timeout`/`exception` 显中文非生串、同主机同 status 合一组）、`target_unreachable` 细化到 kind、未知 kind 回退「不可达」、`failed==0` 不渲染、Lark **四**种逗号排列（`failed>0`∧无 finding / `failed>0`∧有 finding / `failed>0`∧meta-None / `failed==0`）均合法 JSON、Telegram meta-None∧failed>0 抬头后直渲、MarkdownV2/`tojson` 转义。
